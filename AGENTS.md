# Codex 开发规则

## 目标
实现可回测、可解释、可迭代的蓝莓加工峰值预测 Agent。核心预测由统计/机器学习代码完成，LLM 不得自行编造吨数。

## 不可违反的业务规则
1. 普通用户默认只提交农场位置和各品种定植亩数；农场面积参与产量估算，加工厂建筑面积不得作为峰值预测特征。
2. 默认仅使用1—4月数据。
3. 默认剔除：普鲜、普青、普冻、废果。
4. 默认剔除：巴松加工厂。
5. 单日最大值与连续3日持续峰值必须同时输出。
6. 春节窗口不得直接删除用于所有模型；应保留原始事实，并在自然成熟模型与到果实现模型之间分层处理。
7. 所有预测必须记录：数据版本、模型版本、训练截止日、预测生成时间、输入快照和置信区间。
8. 任何模型上线前必须通过跨产季、滚动时间、加工厂留出三类回测。
9. 禁止随机拆分相邻日期作为主要验证方法，避免时间泄漏。
10. 历史原始表不可覆盖或修改，使用 append-only raw 层。
11. 亩产、商品果率、成熟曲线和采摘实现率的自动推断必须来自版本化参数库；不得在接口代码中硬编码经验数字。
12. 极简输入模式不得因为缺少高级参数而拒绝预测；应使用分层先验并扩大区间，同时展示假设、来源和可信度。
13. 输出必须包含逐日P50/P80/P90、单日峰值、持续峰值、产季曲线、详细计算说明和规则化建议。
14. LLM仅组织说明和调用工具，所有吨数、区间、贡献率和建议触发条件由确定性服务返回。

## 工程要求
- 每项任务先阅读 `docs/` 对应规范。
- 先写测试，再实现核心规则。
- 所有数据库变更使用 Alembic。
- 所有清洗规则配置化，不在业务代码中散落硬编码。
- API 返回 Pydantic 类型；金额/重量避免浮点累计误差，数据库使用 NUMERIC。
- 导入必须幂等：相同文件重复导入不会产生重复事实记录。
- 日志中不得记录整行敏感业务数据。
- 所有模型结果必须可复现：固定随机种子，保存特征清单和参数。

## 完成定义
- 代码通过格式检查、类型检查和 pytest。
- 有最小可运行示例。
- 有回测报告 JSON 和 Markdown。
- 预测结果可追溯到输入行和模型版本。
